# 原型设计与代码规范

## 1. 设计目标

保持当前仓库的紧凑型企业级中后台风格，用于高密度业务评审。页面应优先保证业务字段完整、状态清晰、跨页链路可演示，不进行无业务依据的视觉重设计。

## 2. 当前视觉基线

| 项目 | 当前规范 |
| --- | --- |
| 顶部栏 | 56px 固定顶部栏，蓝色渐变，右侧展示环境、角色和用户 |
| 左侧导航 | 220px 固定侧栏，当前项浅蓝背景和蓝色指示条 |
| 内容区 | 浅灰蓝背景，白色卡片，紧凑间距 |
| 字体 | 系统字体、PingFang SC、Microsoft YaHei，正文约 14px |
| 主色 | `#4c78f5` / Ant Design 蓝色系 |
| 状态色 | 成功绿、警告橙、危险红、信息蓝、禁用灰；必须同时有文字 |
| 圆角 | 以 4–6px 为主，避免消费端大圆角 |
| 表格 | 高信息密度、浅色横向分隔、行 hover、操作列固定、必要时横向滚动 |
| 表单 | 标签与字段尽量同一行，桌面端 3–4 列，窄屏降为 2 列或单列 |
| 反馈 | 短结果用 toast/message，危险动作使用二次确认，表单错误就近展示 |

## 3. 公共样式复用

### 3.1 基础壳层

- `area-survey-detail.css`：正常任务、角色任务、填报、站点明细和统计页的主要公共壳层。
- `area-survey-outplan.css`：计划外管理列表、创建和详情的另一套基础壳层。

两者含有重复的顶部栏、侧栏、卡片、表格、按钮和标签规则。新增页面应优先选择与入口链路一致的一套，不复制第三套；后续若要合并，需单独做公共样式重构方案和全页回归。

### 3.2 业务公共层

- `area-survey-role-tasks.css/js`：计划内片区所和普查人员任务列表公共实现。
- `area-survey-survey-fill.css`：三类填报页的楼栋、附件、汇总、弹窗和粘底操作区。
- `area-survey-user-fill.css`：用户站报告、未在网和 PDF 模拟层；对公用户在其上叠加 `area-survey-corporate-fill.css`。
- `area-survey-station-details.css`：组织树、查询区、站点宽表、同步/退回弹窗；未在网统计复用其基础表格并叠加页面样式。
- `area-survey-outplan-modal.css`：计划外范围选择与临时用户站弹窗。

## 4. 可复用组件模式

原型虽未引入 Ant Design，但交互与命名应对应以下组件：

| 现有模式 | Ant Design 对应 | 使用约束 |
| --- | --- | --- |
| `.topbar` / `.sider` / `main` | `Layout`、`Header`、`Sider`、`Content` | 所有主页面保持一致尺寸 |
| `.menu-item` | `Menu.Item` | 新入口需同步所有正式主导航页面 |
| `.breadcrumb` | `Breadcrumb` | 入口、当前页和返回链接一致 |
| `.card` | `Card` | 卡片间距紧凑，不嵌套过深 |
| `.filters` | `Form` + `Input/Select/DatePicker` | 查询条件点击“查询”后生效；重置恢复默认 |
| `.table-wrap` | `Table` | 固定操作列，宽表横滚，不删业务列 |
| `.tag` | `Tag` | 状态文字统一走字典 |
| `.modal-mask/.modal` | `Modal` | 危险操作二次确认，提交防连点 |
| `.alert` | `Alert` | 风险、范围和临时对象说明 |
| `.pagination` | `Pagination` | 显示总数和当前页 |
| `.toast` | `message` | 仅用于短时结果，不承载长错误明细 |
| 时间线 | `Timeline` | 审核和操作记录按倒序展示 |

## 5. 页面结构规范

1. 顶部栏与正式左侧菜单。
2. 面包屑。
3. 页面标题、副标题、数据范围或来源标签。
4. 查询卡片（如适用）。
5. 主业务卡片、表格或表单。
6. 分页/空状态。
7. 页面级操作区或粘底操作区。
8. 弹窗和 toast 容器。
9. 先加载公共脚本，再加载页面脚本。

页面标题区需持续展示任务来源、计划外原因、轮次等关键上下文，避免用户在复用页面中迷失。

## 6. HTML/CSS 规范

- 使用语义清晰的 `header/aside/nav/main/section/table`，保持可直接双击运行。
- 页面特有类名使用页面或业务前缀，避免全局覆盖 `.card`、`.btn` 等公共类。
- 优先补充公共 CSS 文件，不在 HTML 中堆叠大量 `style=`；当前仓库仍有较多列宽内联样式，新增业务样式不继续扩大。
- 表格单元格默认不随意换行；长文本用 `title`、省略号或详情查看。
- 固定操作列需为表头和内容分别设置正确层级与 hover 背景。
- 控件、标签和按钮在键盘焦点、禁用、错误、加载状态下应可辨识。
- 状态不能只靠颜色表达；危险、成功、等待必须有明确文字。
- 响应式优先保证 1280px 及以上桌面评审，窄屏表单降列，宽表保留横向滚动。

## 7. JavaScript 规范

- 业务枚举集中定义并通过显示字典转换，不在多个页面散落硬编码。
- 新增函数按“读取/标准化 → 派生 → 校验 → 持久化 → 渲染 → 反馈”分层，避免单函数同时拼接全部流程。
- 不直接信任 URL JSON 或 localStorage；解析使用 `try/catch`，执行字段缺省和枚举标准化。
- 跨页只传业务 ID、来源 `source`、动作 `audit` 等最小上下文；避免继续传完整业务对象。
- 写入 localStorage 后再导航；不能依赖返回页读取 `taskUpdate` 才完成持久化。
- 新增存储结构需带 `schemaVersion`，提供旧数据迁移和未知值保留策略。
- 任务、对象、轮次、项目、附件使用稳定 ID；显示序号用渲染时 `index + 1`，不写入业务数据。
- 关键动作需防重复提交，检查当前状态并记录操作日志。
- 不用 `innerHTML` 拼入未经转义的用户输入；当前原型已有相关风险，新代码应提供统一 `escapeHtml()`。

## 8. 状态与数据规范

- 流程节点、审核结果、下发状态、轮次有效性、建卡状态、同步状态分字段保存。
- 标准编码稳定，中文显示文案通过字典管理；历史别名只在读取迁移层处理。
- 同一业务对象只能有一个事实源，列表统计从对象子任务/有效轮次派生，不从多个草稿 key 猜测。
- 临时对象与正式对象使用互斥引用；不得把 `tmp-*` 当正式站点 ID。
- 任务内临时对象不更新正式档案；后续建档通过映射记录关联。
- 同站点同年度多轮必须保留历史；新轮次通过前不得提前取消旧有效轮次。

## 9. 路由与导航规范

- 所有链接使用相对路径，不依赖外部网络。
- 每页至少验证：正常入口、刷新、缺少参数、非法 ID、返回按钮、面包屑、左侧菜单。
- 复用填报/详情页必须传 `source`，并保证返回原业务链路。
- 新页面加入正式左侧菜单时，需同步所有正式导航页面；遗留页不作为扩展入口。
- 链接目标和菜单选中项必须与实际来源一致；禁止使用 `source=office` 代替计划外来源。

## 10. 校验与验收

每次修改至少执行：

1. `git diff --check`；
2. 所有改动 JS 的 `node --check`；
3. 改动 HTML 的内联脚本静态解析；
4. 静态 `href/src` 目标检查；
5. 浏览器验证本次入口、主路径、异常路径、返回路径和 localStorage 更新；
6. 桌面常用宽度下检查表单对齐、宽表横滚和固定操作列；
7. 更新页面清单、项目变更记录及相关状态/数据文档。

如无法完成浏览器验证，交付时必须明确写“仅完成静态校验”，不能表述为页面已完整验收。
